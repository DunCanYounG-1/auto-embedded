# 板间通信协议与总线可靠性（UART/CAN/RS485/SPI/I2C：帧 · CRC · 超时重传 · 心跳）

> 本文件在以下情形按需加载：
> - 多板/多模块机器人要互传数据（主控↔电机板↔传感器↔上位机/OpenMV）
> - 自己设计串口/CAN 帧协议，或出现"丢包/粘包/串扰/总线挂死/偶发错乱"
>
> 这是机器人栈的**通信骨架**。调试工具见 tool-skills（`can-debug`/`modbus-debug`/`serial-monitor`）；命令行见 `cli-command-framework.md`。

---

## 铁律

**总线会丢包、会误码、会有人掉线——协议必须自带"边界 + 校验 + 超时 + 失联处理"，否则现场偶发错乱无法定位。任何收到的帧先验证再用；任何依赖的对端都要有心跳与失联降级。**

- **粘包/半包是常态**（流式总线 UART 尤其）：必须有帧边界（帧头+长度 或 分隔符），别假设"一次收到一整帧"。
- 失联要进**安全降级**（如遥控丢失→停车/悬停，呼应 `control-safety-limits.md`），不是放任。

## 选型（按场景）

| 总线 | 拓扑/距离 | 速率 | 抗扰 | 适用 |
|---|---|---|---|---|
| **UART** | 点对点(/弱多点) 短 | 中 | 一般 | 主控↔模块/上位机/OpenMV、调试 |
| **RS485** | 多点 半双工 长(km) | 中 | 强(差分) | 长距离/多设备(Modbus) |
| **CAN** | 多主 总线 中 | 1Mbps(FD更高) | **强(差分抗共模 + 错误检测后硬件自动重发；多主无损仲裁另算可靠性)** | 分布式、噪声大的底盘/多电机（RoboMaster 用法）|
| **SPI** | 主从 板内 短 | 高(MHz) | 弱(无校验) | 板内高速器件(传感器/Flash/编码器) |
| **I2C** | 主从 板内 短 | 低-中 | 弱(易锁死) | 板内低速多器件；硬件 I2C 有锁死风险需超时恢复 |

> 噪声大、多电机分布式 → **CAN 首选**（差分抗扰、仲裁无损、硬件自动重发）。长距离多设备 → RS485。板内 → SPI/I2C。

## 帧协议设计（自定义串口/无线帧）

一帧六字段（五个协议要素 + 载荷）：

```
[帧头] [地址/ID] [功能码/类型] [长度] [载荷...] [校验(CRC)]
 同步边界  发给谁    这帧干啥     防越界   实际数据    检错
```

- **帧头**：选不易与数据冲突的魔数（如 `0xAA 0x55`）；或用 **COBS** 编码消除帧内分隔符冲突（流式更稳）。
- **长度字段**：明确载荷长度，配合帧头做"帧头+长度"切包，解决粘/半包。
- **校验**：短帧用校验和/CRC8，重要/长帧用 **CRC16**（查表实现快）。收帧先校验，错则丢弃（可计错误率）。
- **序号 SEQ**：检测丢包/乱序；要可靠则配 ACK + 超时重传（注意重发幂等）。
- **解析用状态机**（等帧头→收长度→收载荷→校验），逐字节喂入，配 DMA+空闲中断收（见 `realtime-scheduling-isr-dma.md`）。

## CAN 要点

- **标识符 ID = 优先级 + 寻址**：ID 越小优先级越高（仲裁无损）；规划好各节点/消息 ID 段（参考 CiA / RoboMaster 电机 ID 约定）。
- **位定时**：所有节点波特率/采样点一致，否则偶发错误帧；长总线两端 120Ω 终端电阻。
- **bus-off 恢复**：连续错误会进 bus-off，要检测并自动恢复（重初始化/等恢复），否则总线"挂死"。
- 用**过滤器**只收关心的 ID，减负载；高频数据控制周期与带宽匹配（别撑爆总线）。

## 可靠性机制（按需加）

| 机制 | 解决 | 做法 |
|---|---|---|
| 帧边界 | 粘包/半包 | 帧头+长度 / COBS / 分隔符 |
| CRC | 误码 | 收帧校验，错即丢 |
| 超时 | 对端无响应 | 每请求设 deadline，超时重试/报错 |
| 重传 + ACK + SEQ | 丢包 | 序号去重、有限次重发、幂等 |
| **心跳** | 掉线/卡死 | 周期发"我还在"，对端超时未收→判失联→降级 |
| 限流/背压 | 拥塞 | 控制发送频率，匹配带宽与处理能力 |

## 常见陷阱

| 陷阱 | 说明 |
|---|---|
| 假设一次收一整帧 | 粘/半包→错乱；必须帧边界+状态机切包 |
| 收帧不校验直接用 | 误码当真值→行为诡异且难复现 |
| 无心跳/失联处理 | 对端掉线主控不知→失控（要降级到安全态）|
| CAN 不处理 bus-off | 错误累积→总线挂死 |
| 硬件 I2C 无超时恢复 | 锁死后全线卡住；加超时+总线复位（见 imu-gyroscope-checklist）|
| 阻塞式收发在控制环里 | 拖慢/卡死控制；用 DMA+中断+环形缓冲 |
| 魔法数字帧头与数据撞 | 误同步；选稀有魔数或 COBS |
